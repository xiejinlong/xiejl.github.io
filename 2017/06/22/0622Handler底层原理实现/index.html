handler底层原理详解

发表于 2017-06-22

handler成员

在android中，线程切换基本全都是通过handler，不过利用了什么控件，多么简单的线程切换方式，也是通过封装handler来实现的。

Message

Message是handler里面传递的元素，这个类会实现parcelble接口，我们看看Message定义的一些参数变量。

  • what:这个是我们用来确认消息的类型，在handleMessage里面处理时需要通过这个what去判断
  • arg1:可以携带一个参数arg1，我们也可以通过将arg设置到bundle里面，
  • arg2:可以携带一个参数arg1，我们也可以通过将arg设置到bundle里面
  • obj: 我们让Message携带一个OBJ对象
  • replyTo：如果是使用Messenger，这个会标记Messenger对象
  • data：这个是一个bundle类型的，
  • target：这个是handler类型，标记了这个Message属于哪个handler
  • callback： 消息携带的Runnable对象
  • next：记录了下一条消息

这个Message会有一个消息池，默认的池的大小是50。可以通过recyler来进行回收清空。


对于我们创建出Message的方式有两种，一种是直接new出来，另外一种是通过obtain获取消息池的消息,实际上在我们new的时候，内部同样也是调用的obtain方法去获取。

MessageQueue

在handler的使用中，我们很少直接和MessageQueue接触，在我们将消息post出去之后，消息将会在MessageQueue中存储。我们来看看handler到MessageQueue的入口。

Looper

looper作为一个连接器，一个线程中只会有一个looper，一个looper中也只会有一个消息队列。我们先来看看looper的创建。Looper的创建方式并非直接通过new创建，而且通过prepare方法创建。在ActivityThread创建时便会创建出来相应的Looper


在myLooper中，会通过ThreadLocal获取当前线程的looper，所以这个Looper也是以线程划分的。
在创建出来了looper之后，会调用Looper.looper()方法开始循环读取消息：

在这个looper方法中，我们主要是关注两个方法，queue.next()和dipatchMessage(),其中queue.next会从消息队列中获取出下一个消息，dispatchMessage会用来处理消息。

其实handler的线程切换只是调换位置的切换而已，因为handler是作为当前类的局部变量，也就会被当前线程所持有，当调用了这个handler的sendMessage的时候。会被当前线程的looper所读取出来，并且调用handleMessage处理。

handler使用

其实handler的使用非常简单，分为以下三个部分。

  • 定义handler，实现handleMessage方法

    Handler handler = new Handler() {

        @Override
    public void handleMessage(Message msg) {
        //deal with message;
}
  • 在想要的地方发送消息
    handler.postMessageDelay().
    handler.postMessage();

    handler之JNI

MessageQueue的jni代码对应

  • nativeInit()

  • nativePollOnce

  • nativeWake

  • nativeIsPolling

  • nativeSetFileDescriptorEvents

通过上面几个MessageQueue的jni代码,我们可以明显的看出来，它并不是具体的操作实体，它会将操作再次转交给looper。这个looper并不是所谓的java层的Looper，他是JNI的结构体ALooper。

ALooper及Looper类

  • pollOnce

在这里，又会调用looper的pollOnce:

  • pollAll 读取所有消息
    内部通过循环调用pollOnce，读取每一条消息

  • sendMessage 发送消息

事实上，handler机制就是通过自线程持有外部线程的handler引用，然后调用该引用的handler的postMessage方法，这样就会将message发送到这个线程的消息队列里面了。这个message内部有个target标记了所属的hanlder对象，这样保证了一个线程中可以存在多个handler对象。在发送时，底层通过线程间通信，通过管道传递消息，因为管道分为两种，其中一种就是用与父子线程的传递，消耗的资源较少。而Looper在取消息时，因为消息中就存储了target Handler的引用，那么就可以调用这个handler的handleMessage方法，并且处理将这个消息携带过去处理。
